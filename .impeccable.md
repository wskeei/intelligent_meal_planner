# Intelligent Meal Planner Design Context

## Design Context

### Users
- Primary users are Chinese-speaking consumers who want a practical daily meal plan, not a nutrition dashboard.
- Typical usage happens on mobile during short planning windows such as commuting, grocery prep, or deciding what to eat today.
- Users need the product to reduce decision effort, explain what information matters, and avoid making health or budget claims it cannot support.
- Primary job to be done: Users should be able to tell the system their constraints and get a trustworthy daily meal plan without guessing what the system needs next.

### Usage Context
- The shortest successful path is `登录/注册 -> 资料补全 -> 对话 intake -> 查看结果 -> 继续/复用`.
- First-time users should be able to start with minimal personal data, then fill in sensitive body metrics only when the planner needs them.
- Returning users should immediately see whether they can continue, reuse a prior setup, or fix a missing prerequisite.

### Brand Personality
- Trustworthy, plainspoken, supportive.
- The interface should feel like a calm nutrition assistant that helps users move forward, not like an AI showcase or admin console.
- Copy should emphasize what the user can do next, what is already known, and why specific data matters.

### Aesthetic Direction
- Product direction is practical, clean, and mobile-first, with restrained green accents and soft neutral surfaces.
- Visual emphasis should come from hierarchy, spacing, and state clarity rather than decorative gradients or novelty.
- Supporting surfaces can feel warm and polished, but the primary experience should remain utility-first and easy to scan.

### Aha Moment
- The product becomes convincing when a user sees that the system understands their goal, budget, and missing inputs, then turns that into a concrete budget-safe daily plan with an obvious next step.

### Anti-References
- Do not make this feel like an AI demo, a speculative startup landing page, or a dashboard full of equal-priority cards.
- Avoid fake affordances, hover-only actions, decorative product jargon, and mixed-brand chrome.
- Avoid language that celebrates the system more than the user outcome, including generic "Powered by AI" framing in primary product surfaces.

### Design Principles
1. Truth before flourish: every CTA and status must match real behavior.
2. One dominant next action: each screen should make the next useful step obvious within seconds.
3. Explain missing information: tell users what is missing, why it matters, and how to resolve it.
4. Lower cognitive load: remove redundant choices, duplicate actions, and decorative dashboard framing.
5. Mobile-safe by default: no critical hover behavior, no horizontal overflow, and touch targets sized for phones.

### Remediation Priorities
1. Trustworthy behavior
2. Clear first-run path
3. Lower cognitive load
4. Consistent, mobile-safe presentation
