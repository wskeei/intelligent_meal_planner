# 对话式智能配餐改造设计

## 1. 背景

当前系统的配餐入口建立在显式营养参数之上：

- 前端 `MealPlanView.vue` 通过滑块要求用户填写热量、蛋白质、碳水、脂肪和预算。
- 后端 `MealPlanRequest` / `UserPreferences` 也以这些数值字段作为主输入。
- 现有 “AI 模式” 只是把一句自然语言转换成这些显式数值，再走旧的 RL 配餐链路。

这带来两个核心问题：

1. 大多数用户并不知道自己准确需要多少热量和宏量营养素。
2. RL 模型难以稳定满足所有显式约束，容易出现某项超标或预算边缘超支的问题。

本次改造的目标，是把系统从“用户手动给数值，算法尽量凑”改为“营养师式对话采集需求，系统内部隐式量化，再进行预算硬约束配餐”。

## 2. 目标与非目标

### 2.1 目标

- 删除现有手动设置热量、蛋白质、碳水、脂肪、预算的配餐入口。
- 系统只保留“多轮对话式”配餐入口。
- DeepSeek 扮演营养师，通过多轮追问采集用户信息。
- 用户公开对话中不再要求填写具体热量和宏量营养数值。
- 系统内部允许把模糊需求映射成隐藏数值目标，以便继续复用 RL 配餐能力。
- 用户预算是硬约束，最终输出的正式配餐结果必须不超过预算。
- 如果当前预算无法满足需求，系统直接回复“预算过低，需要提高预算”，不返回超预算方案。
- 如果用户档案中缺失 `性别/年龄/身高/体重/运动量`，AI 在对话中补问并自动写回用户资料。

### 2.2 非目标

- 不重做 RL 训练体系与奖励体系的整体方法论。
- 不在本阶段引入疾病饮食处方、医学诊断或临床级建议。
- 不实现开放式长期聊天机器人；本次只实现围绕“生成一日三餐配餐方案”的有限状态对话。

## 3. 强约束

本方案必须满足以下强约束：

1. 产品层面只保留对话式入口，旧滑块式入口彻底移除。
2. 对话过程采用“逐步追问，信息够了再一次性正式生成”的模式。
3. 预算是唯一对外显式的硬输入，且必须被严格遵守。
4. 系统内部数值目标对用户不可见，不在对话引导中展示。
5. 用户资料优先从账户读取；只有账户缺失时才通过对话补齐。
6. AI 对话中收集到的身体资料要回写到数据库，下次无需重复询问。

## 4. 总体方案

采用“状态机对话 + LLM 辅助抽取 + 硬约束预算守卫”的架构。

### 4.1 核心思想

- `DeepSeek` 不直接决定整个流程，而是只负责辅助理解用户自然语言。
- 后端维护一个确定性的会话编排器，明确知道当前缺什么信息、下一轮该问什么、何时可以生成方案。
- 身体资料和配餐偏好统一沉淀为结构化会话状态。
- 当信息齐备后，后端通过确定性规则计算隐藏营养目标，再调用 RL。
- RL 推理必须启用“严格预算模式”，任何超预算结果都视为失败。

### 4.2 功能角色

为了兼顾论文中的“多角色协作”表述与工程可靠性，运行时组件划分为以下职责单元：

- `Profile Resolver`
  - 从当前登录用户读取身体资料。
  - 判断哪些字段缺失。
  - 接收 AI 对话收集到的新字段并回写数据库。

- `Conversation Orchestrator`
  - 管理会话状态。
  - 决定下一轮问题。
  - 判断何时达到“可生成配餐”的条件。

- `DeepSeek Slot Extractor`
  - 从用户自然语言中提取预算、目标、忌口、口味偏好和可能的身体资料更新。
  - 只返回结构化 JSON，不直接驱动业务流程。

- `Hidden Target Mapper`
  - 根据身体资料 + 健康目标计算内部热量与宏量目标。
  - 对数值范围做裁剪，保证不偏离 RL 模型训练区间。

- `Budget Guard`
  - 在 RL 推理前做一次预算可行性预检查。
  - 在 RL 推理后做一次严格预算复核。

- `Strict-Budget RL Planner`
  - 在严格预算掩码下调用现有 RL 模型生成候选配餐。
  - 如中途没有预算内可选动作，则立即终止并返回失败。

## 5. 用户流程设计

### 5.1 首次进入配餐页

用户进入 `/meal-plan` 页面后，前端自动创建一个新的“对话式配餐会话”，并展示营养师欢迎语。

欢迎语只介绍能力边界，不暴露内部数值目标，例如：

> 我会先了解你的身体情况、当前目标、预算和口味偏好，再帮你整理一份预算内的一日三餐方案。

### 5.2 对话顺序

系统按以下优先级逐步追问：

1. 账户缺失的身体资料
   - 性别
   - 年龄
   - 身高
   - 体重
   - 活动强度

2. 配餐关键信息
   - 当前健康目标
   - 可接受预算
   - 忌口 / 过敏 / 不吃的食物
   - 口味偏好 / 饮食倾向

3. 信息确认
   - 如果上述信息已足够，直接进入正式生成，不再继续闲聊。

### 5.3 结束条件

满足以下条件后，系统进入“正式生成”：

- 身体资料完整。
- 存在明确预算。
- 存在明确健康目标，或可从个人资料默认目标继承。
- 已完成至少一次饮食限制类询问（即使答案是“没有忌口”）。

### 5.4 预算不足处理

生成前若预算预检查失败，或 RL 严格预算推理无法产出预算内方案，则系统直接返回：

> 按你当前的需求，这个预算过低，需要适当提高预算后我再为你正式配餐。

此时会话状态记为 `budget_rejected`，不生成正式配餐结果。

## 6. 隐藏目标映射设计

### 6.1 身体指标计算

使用 Mifflin-St Jeor 公式估算基础代谢：

- 男性：`10 * 体重(kg) + 6.25 * 身高(cm) - 5 * 年龄 + 5`
- 女性：`10 * 体重(kg) + 6.25 * 身高(cm) - 5 * 年龄 - 161`

再结合活动系数得到 TDEE：

- `sedentary`: 1.2
- `light`: 1.375
- `moderate`: 1.55
- `active`: 1.725
- `very_active`: 1.9

### 6.2 健康目标到热量目标的映射

- `lose_weight`: `TDEE - 500`
- `gain_muscle`: `TDEE + 300`
- `maintain`: `TDEE`
- `healthy`: `TDEE`

### 6.3 健康目标到宏量比例的映射

- `lose_weight`: 蛋白 35%，碳水 30%，脂肪 35%
- `gain_muscle`: 蛋白 30%，碳水 45%，脂肪 25%
- `maintain`: 蛋白 30%，碳水 40%，脂肪 30%
- `healthy`: 蛋白 25%，碳水 45%，脂肪 30%

### 6.4 与 RL 训练区间对齐

为了避免内部目标超出既有模型的训练分布，生成后的隐藏目标必须裁剪到相对稳定区间：

- 热量：`1200 ~ 3000`
- 蛋白质：`60 ~ 220`
- 碳水：`50 ~ 400`
- 脂肪：`30 ~ 120`

这些数值只用于后端推理和预算校验，不在前端展示，也不在 AI 追问中显式提及。

## 7. 预算硬约束设计

预算硬约束分为三层：

### 7.1 生成前预检查

使用现有 `feasibility_service` 对隐藏目标进行预算下限预估。

如果在当前预算下连基础可行性都无法达到，则不进入 RL 推理，直接返回预算过低。

### 7.2 RL 严格预算推理

需要修改 `MealPlanningEnv.action_masks()` 的推理逻辑：

- 当前推理允许 `10%` 预算缓冲。
- 当前预算枯竭时会兜底放行“最便宜菜品”，这可能导致超预算。

改造后新增 `strict_budget=True` 推理模式：

- 严格预算模式下不允许任何预算缓冲。
- 如果当前剩余预算下无可选动作，直接返回失败，不再启用超预算兜底。

### 7.3 生成后复核

即使 RL 已经在严格预算模式下运行，后端仍要对最终结果做一次确定性复核：

- `total_cost <= user_budget` 才允许正式返回。
- 否则统一视为预算不足，返回预算过低提示。

## 8. 数据模型设计

### 8.1 现有用户表

当前 `users` 表已经具备本次方案需要的身体资料字段：

- `age`
- `gender`
- `height`
- `weight`
- `activity_level`
- `health_goal`

因此不新增用户身体字段，只补齐“读取完整资料”和“更新资料”的 API 能力。

### 8.2 新增对话会话表

新增 `meal_chat_sessions`：

- `id`: 会话 ID
- `user_id`: 所属用户
- `status`: `collecting_profile | collecting_preferences | budget_rejected | completed`
- `collected_slots`: JSON，保存预算、忌口、口味偏好、目标覆盖等
- `hidden_targets`: JSON，保存内部量化目标
- `final_plan`: JSON，保存最终正式配餐结果
- `created_at`
- `updated_at`

### 8.3 新增对话消息表

新增 `meal_chat_messages`：

- `id`
- `session_id`
- `role`: `assistant | user`
- `content`
- `stage`
- `created_at`

这样可以支持：

- 页面刷新后恢复会话
- 历史对话回显
- 从已完成会话中提取正式配餐结果

## 9. API 设计

### 9.1 认证与用户资料

- `GET /api/auth/me`
  - 返回完整用户资料，而不是当前仅返回 `id/username/email/age/health_goal`。

- `PATCH /api/auth/me/profile`
  - 更新用户身体资料与默认健康目标。
  - AI 在对话中采集到的新身体资料通过该链路持久化。

### 9.2 对话式配餐

- `POST /api/meal-chat/sessions`
  - 创建新会话并返回第一条营养师消息。

- `GET /api/meal-chat/sessions/{session_id}`
  - 获取会话详情与完整消息列表。

- `POST /api/meal-chat/sessions/{session_id}/messages`
  - 发送一条用户消息并返回下一轮 AI 回复。
  - 若本轮已达到生成条件，返回正式配餐结果。

### 9.3 配餐历史

- 保留 `GET /api/meal-plans`
  - 但数据来源从旧的内存 `_history` 调整为已完成会话里的 `final_plan`。

### 9.4 删除旧接口

以下旧接口从运行时逻辑中彻底移除：

- `POST /api/meal-plans` 的手动偏好入参模式
- `POST /api/meal-plans/quick`
- `GET /api/meal-plans/feasibility`

## 10. 前端设计

### 10.1 配餐页

`MealPlanView.vue` 重构为单一聊天页面：

- 删除手动模式 / AI 模式 Tab
- 删除所有热量与宏量滑块
- 删除预算可行性提示条
- 页面只保留：
  - 聊天消息流
  - 输入框
  - 发送按钮
  - 生成中的加载状态
  - 最终正式配餐结果卡片

### 10.2 结果展示

结果展示应从“数值达标百分比面板”调整为“营养师结论 + 菜品清单 + 预算结果”：

- 强调：方案符合当前预算
- 强调：整体饮食倾向，例如“偏高蛋白”“更适合减脂期”“口味较清淡”
- 保留菜品名称和价格
- 若保留营养信息，也应降级到次要区域，不再以“达标率”做主视觉

### 10.3 个人资料页

`ProfileView.vue` 需要切换为“服务端资料为准”：

- 当前本地 `Pinia + localStorage` 的资料不能再作为唯一数据源。
- 页面应展示并编辑后端真实资料。
- AI 对话补全资料后，用户返回个人资料页应能直接看到更新结果。
- 当前个人中心的“每日热量 / 宏量目标展示”建议移除或弱化，避免与“前台不暴露数值目标”的策略冲突。

### 10.4 首页

首页当前显著展示“每日目标热量”，这会继续强化旧心智。建议改为展示：

- 当前健康目标
- 个人资料完整度
- 进入营养师对话式配餐的主入口

## 11. 配置设计

根目录 `.env` 由用户填写 DeepSeek 信息，后端读取：

```env
DEEPSEEK_API_KEY=
DEEPSEEK_API_BASE=https://api.deepseek.com/v1
DEEPSEEK_MODEL=deepseek-chat
```

前端如需覆盖 API 地址，继续使用：

```env
VITE_API_BASE_URL=http://127.0.0.1:8000/api
```

## 13. 风险与对策

### 风险 1：LLM 抽取 JSON 不稳定

对策：

- DeepSeek 只做“结构化提取”，不直接决定业务流程。
- 后端对抽取结果做字段级验证与默认值兜底。

### 风险 2：预算可行性预估不精确

对策：

- 保留预算预检查作为提前拦截。
- 真正的硬保证由 `strict_budget` 推理和后置复核共同完成。

### 风险 3：前后端资料源不一致

对策：

- 把后端用户资料作为唯一真源。
- Pinia 只做会话内缓存，不再独立计算并长期持久化本地画像。

## 14. 验收标准

当以下条件全部满足时，认为改造完成：

1. 前端已不存在热量、蛋白质、碳水、脂肪、预算滑块式配餐入口。
2. 用户进入配餐页后只看到营养师对话界面。
3. 缺失身体资料时，系统会逐项追问并自动回写数据库。
4. 用户无需知道具体营养数值，也可以完成正式配餐。
5. 最终正式配餐结果不会超出用户预算。
6. 预算不足时，系统明确回复需要提高预算，不返回超预算方案。
7. 已完成会话能够在历史页中回看正式配餐结果。
